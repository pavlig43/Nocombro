# UI Flow Testing Summary

Дата: 2026-04-18

## К чему пришли

- Основной упор имеет смысл делать не на unit-тесты, а на flow/integration тесты.
- Для проекта естественны сценарии уровня UI:
  - открыть приложение
  - перейти на экран
  - создать сущность
  - сохранить
  - открыть снова
  - изменить
  - проверить результат на экране и в БД
- Тесты должны работать с настоящей Room-схемой и настоящей тестовой БД.

## Какие типы тестов нужны

### 1. CI-friendly flow tests

Для GitHub/CI подходят:
- Compose UI tests через семантику
- поиск элементов по `testTag`, тексту и другим Compose semantics
- проверки состояния UI
- проверки итогового состояния БД

Это не "моки ради моков", а сценарии на почти реальной сборке приложения.

### 2. Локальные более тяжелые прогоны

Локально можно гонять:
- более длинные smoke/regression сценарии
- сценарии на большом реалистичном датасете за год
- ручные exploratory-прогоны

## Что не является целевым путем

- не делать ставку на массовые unit-тесты
- не рефакторить архитектуру ради абстрактной "тестируемости"
- не тащить OS-level record/replay и прочую нестабильную механику

## Какие библиотеки релевантны

Базовый набор:
- `Compose UI Test`
- `Kotest` или `JUnit`
- `kotlinx-coroutines-test`
- `test:database-kit`

Дополнительно, только где реально нужно:
- `Turbine` для проверки `Flow`
- `MockK` для редких локальных подмен
- `Koin Test` если понадобится тестово собирать DI-граф

## Пример целевого flow

Один из первых полезных сценариев:

1. Открыть приложение
2. Перейти на вкладку продуктов
3. Создать продукт
4. Сохранить
5. Убедиться, что продукт виден в списке
6. Открыть его снова
7. Изменить поля
8. Сохранить
9. Проверить экран
10. Проверить запись в БД

## Важная практическая мысль

В тестах нормально вынести в отдельный helper инфраструктурный bootstrap:
- создание тестовой БД
- создание `RootNocombroComponent`
- подачу тестовых `RootDependencies`
- `setContent { App(root) }`

При этом сам flow не надо прятать глубоко:
- шаги сценария должны оставаться читаемыми прямо в тесте

## Что это значит для следующего шага

Когда появятся реальные данные за год, следующий практический шаг:

1. Выбрать 3-5 реальных пользовательских flow
2. Определить, какие из них идут в CI
3. Определить, какие `testTag` нужны для этих flow
4. Подготовить test harness для запуска `App(root)` на тестовой БД
