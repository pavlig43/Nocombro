# План приемочного тестирования Nocombro на годовых данных

## Назначение

Этот документ описывает полный план приемочного тестирования после загрузки реальных данных фирмы за год.

Цель плана:
- проверить не только техническую стабильность, но и достоверность бизнес-результатов;
- убедиться, что формы, таблицы, storage и аналитика согласованы между собой;
- получить повторяемый сценарий приемки перед дальнейшими доработками и регрессией.

Этот документ дополняет рабочие материалы в `test/`, в первую очередь:
- `test/first-pass-checklist.md`
- `test/real-data-risk-matrix.md`
- `test/real-data-observations-template.md`

Эти файлы помогают провести быстрый первый проход, фиксировать наблюдения и смотреть на приемку через матрицу рисков.
Этот файл отвечает за полный сценарий ручной и полуформальной бизнес-приемки на реальных данных.

## Основная цель приемки

Система считается успешно прошедшей приемку, если:
- данные за год загружены без потерь и явных искажений;
- основные пользовательские сценарии выполняются без падений;
- формы корректно создают, редактируют и повторно открывают сущности;
- таблицы отображают актуальные данные и не теряют строки после фильтрации или сортировки;
- аналитика за месяц, квартал и год совпадает с ручной сверкой по контрольным периодам;
- изменения старых записей корректно пересчитывают связанные экраны;
- после перезапуска приложения данные и настройки остаются согласованными.

## Классификация тестов

### 1. Техническая целостность

Проверяем, что приложение работает стабильно как система:
- приложение запускается;
- основные экраны открываются;
- нет падений при навигации;
- зависимости и компоненты создаются корректно;
- после сохранения данные можно повторно открыть;
- после перезапуска состояние не теряется.

### 2. Функциональная корректность

Проверяем пользовательские сценарии:
- создание сущностей;
- редактирование сущностей;
- повторное открытие существующих записей;
- влияние изменений на таблицы, storage и аналитику;
- корректность фильтров, сортировок и отображения.

### 3. Бизнес-валидация

Проверяем, что цифры и выводы системы отражают реальную деятельность фирмы:
- обороты;
- расходы;
- результаты по периодам;
- profitability;
- остатки и движения в storage;
- влияние исторических исправлений на итоговую картину.

### 4. Регрессия

После каждого исправления повторяем:
- критические smoke-сценарии;
- контрольные сверки по аналитике;
- сценарии, которые уже ломались ранее.

## Наборы данных для проверки

Для надежной приемки нужны четыре рабочих набора:

### `empty`

Пустая база.

Используется для проверки:
- первого запуска;
- пустых экранов;
- поведения аналитики без данных;
- поведения таблиц без строк;
- отсутствия падений в пустом состоянии.

### `smoke`

Минимальный набор данных:
- несколько товаров;
- несколько контрагентов;
- несколько документов;
- несколько транзакций;
- несколько расходов.

Используется для быстрой проверки базовых сценариев после изменений.

### `year-real`

Полная реальная деятельность фирмы за год.

Это основной приемочный набор.
На нем проверяется:
- достоверность отчетов;
- корректность итогов;
- устойчивость интерфейса на большом объеме записей;
- корректность пересчета истории.

### `edge`

Набор с граничными и неудобными случаями:
- похожие названия;
- пустые необязательные поля;
- даты на границе месяцев и года;
- нулевые суммы, если допустимы;
- большие суммы;
- старые записи, которые меняются задним числом;
- плотные серии операций в один день.

## Порядок приемки

Приемка проходит в шесть этапов.

### Этап 1. Подготовка

Перед началом нужно зафиксировать:
- период данных, который загружается;
- список ключевых бизнес-экранов;
- список показателей, которые будут сверяться вручную;
- контрольные периоды для сверки;
- перечень критических сценариев, без которых системе нельзя доверять.

Также стоит заранее определить:
- один слабый месяц по объему операций;
- один средний месяц;
- один пиковый месяц;
- один квартал;
- весь год.

### Этап 2. Базовая стабилизация

После первой загрузки данных проверяем:
- приложение открывается;
- все главные вкладки доступны;
- аналитические экраны открываются;
- таблицы не пустые там, где должны быть данные;
- формы открываются без ошибок;
- нет падений на переходах между основными экранами.

Это минимальный проход, который подтверждает, что можно переходить к предметной приемке.

### Этап 3. Проверка CRUD и связности

На этом этапе проверяем:
- создание новой записи;
- открытие существующей записи;
- редактирование существующей записи;
- повторное открытие после сохранения;
- появление изменений в связанных таблицах;
- появление изменений в storage;
- появление изменений в аналитике.

### Этап 4. Проверка аналитики на реальном годе

Проверяем:
- месяц;
- квартал;
- год;
- пустые периоды;
- пограничные даты.

Каждый ключевой показатель сверяется вручную хотя бы на нескольких контрольных периодах.

### Этап 5. Граничные и негативные сценарии

Проверяем:
- пустой результат фильтра;
- редактирование старых записей;
- похожие названия;
- работа на плотном массиве данных;
- корректность после изменения записей задним числом.

### Этап 6. Регрессия после исправлений

После найденных дефектов повторяем:
- smoke;
- контрольные сценарии по формам;
- контрольные сверки аналитики;
- проблемные кейсы, которые уже проявлялись.

## Приоритеты

### P0

Без этого системе нельзя доверять:
- приложение не падает на основных сценариях;
- формы сохраняют и повторно открывают корректные данные;
- таблицы показывают актуальные данные;
- аналитика считает корректно;
- profitability считает корректно;
- storage отражает ожидаемое состояние;
- редактирование старых данных не ломает итоговую картину;
- после перезапуска данные сохраняются.

### P1

Очень важно для ежедневной работы:
- сортировки и фильтры;
- корректная навигация к сущности по спискам;
- сохранение настроек;
- корректность отображения дат и сумм;
- отсутствие зависаний на больших объемах данных.

### P2

Важно, но не блокирует приемку:
- редкие UX-сценарии;
- косметические проблемы;
- нечастые визуальные несоответствия;
- редкие крайние случаи, не влияющие на итоговые цифры.

## Модульный план проверки

## Формы

### `features/form/document`

Проверить:
- создание документа;
- редактирование документа;
- повторное открытие существующего документа;
- сохранение всех ключевых полей;
- корректность даты;
- появление документа в связанных таблицах;
- влияние на аналитику;
- отсутствие дублирования после повторного сохранения.

Контрольные кейсы:
- документ с минимальным набором полей;
- документ с полным набором полей;
- документ на границе периода;
- изменение старого документа;
- повторное открытие документа по `id`.

Риски:
- неверная дата;
- потеря части полей;
- неправильная связь с другой сущностью;
- аналитика не обновилась после исправления.

### `features/form/product`

Проверить:
- создание товара;
- редактирование названия и ключевых характеристик;
- корректный выбор товара в других формах;
- отсутствие потери связи с историческими данными;
- корректное отображение в таблицах.

Контрольные кейсы:
- похожие названия;
- минимально заполненный товар;
- товар, уже участвующий в операциях;
- редактирование используемого товара.

Риски:
- дубли;
- потеря ссылок;
- искажение исторических данных.

### `features/form/vendor`

Проверить:
- создание контрагента;
- редактирование контрагента;
- выбор контрагента в связанных формах;
- корректное отображение в таблицах и аналитике;
- повторное открытие существующего контрагента.

Контрольные кейсы:
- похожие названия;
- изменение реквизитов;
- контрагент, который уже фигурирует в истории.

Риски:
- дубли;
- неверная привязка документов;
- невозможность корректного редактирования старых записей.

### `features/form/declaration`

Проверить:
- создание декларации;
- редактирование декларации;
- корректность дат и периода;
- влияние на связанные экраны;
- повторное открытие и сохранение без потерь.

Риски:
- потеря связи с базовыми данными;
- двойной учет;
- неправильная датировка.

### `features/form/transaction`

Проверить:
- создание транзакции;
- редактирование суммы, типа и даты;
- корректное попадание в аналитику;
- корректное влияние на итоговые показатели;
- корректное отображение в таблицах.

Контрольные кейсы:
- несколько транзакций в один день;
- транзакции на границе месяцев;
- исправление суммы задним числом;
- повторное открытие старой транзакции.

Риски:
- двойной учет;
- расхождение итогов по периоду;
- неверное влияние на profitability и storage.

### `features/form/expense`

Проверить:
- создание расхода;
- редактирование суммы, категории и даты;
- попадание в аналитику;
- корректное участие в profitability;
- корректное отображение в таблицах.

Контрольные кейсы:
- регулярный расход;
- крупный разовый расход;
- расход на границе месяца;
- изменение категории задним числом.

Риски:
- расход не попал в отчет;
- расход попал в неверную категорию;
- расход учтен дважды.

## Таблицы

Покрываем `features/table/*`.

Проверить:
- отображение всех ожидаемых строк;
- корректность сортировки;
- корректность фильтров;
- корректность поиска, если он есть;
- открытие сущности из строки;
- обновление таблицы после сохранения формы;
- отсутствие пропажи строк после фильтрации и сброса фильтра;
- корректное форматирование дат и чисел.

Сценарии на большом объеме:
- открыть таблицу с годовым объемом данных;
- отсортировать по нескольким ключевым колонкам;
- применить и сбросить фильтр;
- открыть запись из середины большого списка;
- изменить запись и проверить немедленное обновление списка.

Основные риски:
- строковая сортировка вместо числовой;
- устаревшие данные после сохранения;
- потеря части строк;
- неправильная фильтрация по периоду.

## Storage

Покрываем `features/storage`.

Проверить:
- корректность начального отображения;
- влияние документов и транзакций на состояние;
- корректность после редактирования старых записей;
- отсутствие невозможных состояний;
- согласованность со связанными таблицами и аналитикой.

Контрольные сценарии:
- несколько движений по одной сущности;
- изменения задним числом;
- высокая плотность операций;
- сравнение состояния до и после исправления старой записи.

Ключевая сверка:
- состояние в storage должно объясняться историей данных.

## Аналитика

### `features/analytic/main`

Проверить:
- корректность итогов за выбранный период;
- смену периода;
- пустой период;
- пограничные даты;
- влияние исправлений в исходных данных;
- согласованность итогов с таблицами и детализацией.

Контрольные периоды:
- слабый месяц;
- средний месяц;
- пиковый месяц;
- квартал;
- год;
- пустой диапазон.

Обязательная сверка:
- итог на экране должен совпадать с ручной суммой по контрольным данным.

### `features/analytic/profitability`

Проверить:
- корректность формулы;
- состав входящих показателей;
- корректный учет расходов;
- корректный учет доходов или оборота;
- отсутствие двойного учета;
- корректность результата по разным периодам.

Контрольные кейсы:
- прибыльный период;
- убыточный период;
- нулевой период;
- период с редкими записями;
- период с высокой плотностью данных.

Обязательная сверка:
- вручную перепроверить несколько контрольных периодов.

## База и настройки

### `database`

Проверить:
- запись и чтение сущностей;
- целостность связей;
- сохранение после перезапуска;
- выборки по периоду;
- корректность сортировок и фильтров на уровне данных;
- корректность после изменения старой записи.

### `datastore`

Проверить:
- сохранение пользовательских настроек;
- корректность дефолтных значений;
- сохранение состояния после повторного запуска;
- отсутствие неожиданных сбросов.

## Навигация и DI

Покрываем:
- `app/desktopApp`
- `rootnocombro`
- root navigation
- main-tab navigation
- DI wiring

Проверить:
- каждая основная feature открывается из ожидаемого места;
- маршрут открывает правильный экран;
- при открытии на редактирование передается правильный `id`;
- переходы между вкладками не ломают состояние;
- нет ошибок создания компонентов;
- нет проблем с `scope.get()` и root wiring.

Критические симптомы:
- экран существует, но не открывается;
- открывается не тот child;
- запись открывается не та;
- экран пустой без причины;
- падение при создании компонента.

## Негативные и граничные сценарии

Отдельно проверить:
- пустую базу;
- пустой период;
- один элемент в списке;
- очень большой объем записей;
- одинаковые или почти одинаковые названия;
- большие суммы;
- частые изменения старых записей;
- повторные сохранения;
- открытие старых данных из начала года и конца года;
- реакцию аналитики на корректировки истории.

Если бизнес-логика допускает, дополнительно проверить:
- нулевые суммы;
- отрицательные значения;
- частично заполненные записи;
- отмененные или архивные сущности.

## Производительность и устойчивость

На наборе `year-real` дополнительно проверить:
- время открытия главных экранов;
- время открытия аналитики за год;
- время применения фильтра;
- переключение между основными вкладками;
- отсутствие зависаний при работе с большими таблицами;
- корректность обновления после редактирования записи из большого списка.

Если появляются тормоза, фиксировать:
- на каком экране;
- на каком действии;
- на каком объеме данных;
- воспроизводится ли проблема повторно.

## Контрольные сценарии P0

Эти сценарии должны проходить обязательно:
- создать товар;
- создать контрагента;
- создать документ;
- создать транзакцию;
- создать расход;
- повторно открыть каждую сущность и убедиться, что данные сохранились;
- увидеть сохраненные данные в таблицах;
- увидеть влияние данных на storage;
- увидеть влияние данных на аналитику;
- исправить старую запись и убедиться, что зависимые экраны пересчитались;
- сменить период аналитики на месяц, квартал и год;
- перезапустить приложение и убедиться, что данные и настройки не потерялись;
- пройти главные маршруты приложения без ошибок навигации и DI.

## Порядок ручной сверки на реальном годе

Рекомендуемый порядок:

1. Проверить полноту загрузки.
2. Пройти базовый smoke.
3. Проверить формы и повторное открытие записей.
4. Проверить таблицы.
5. Проверить storage.
6. Проверить аналитику по слабому месяцу.
7. Проверить аналитику по среднему месяцу.
8. Проверить аналитику по пиковому месяцу.
9. Проверить квартал.
10. Проверить год.
11. Исправить несколько старых записей и проверить пересчет.
12. Перезапустить приложение и повторить краткий контрольный проход.

## Критерии завершения приемки

Приемка считается завершенной, если:
- все P0-сценарии пройдены;
- по контрольным периодам нет необъяснимых расхождений;
- ключевые суммы сходятся с ручной сверкой;
- редактирование старых данных не ломает текущую картину;
- приложение стабильно работает на полном годовом массиве;
- остается понятный список точечных дефектов, если они не блокируют использование.
